## ステップ9: PJ固有エージェントの作成判断

### 判断基準

PJ固有エージェントを作成するのは、**以下の全てを満たす場合のみ**:

1. **グローバルエージェントではカバーできない** PJ固有のドメイン知識・制約がある
2. **繰り返し使う** 1回きりではなく、PJを通じて何度も必要になる
3. **ツール制限やフックが必要** 汎用エージェントとは異なるツールアクセスやバリデーションが必要

### PJ種別 × 活動タイプ別の判断

#### 開発PJ
- DB操作がある → `db-reader.md`（読み取り専用制限付き）を検討
- 特定のAPIスタイル/規約がある → `api-developer.md`（PJ固有の規約をプロンプトに埋め込み）を検討
- 特殊なテストフレームワーク → `test-runner.md`（PJ固有のテストコマンド・設定）を検討

#### ドキュメント・執筆PJ
- PJ固有の用語集・表記ルールがある → `terminology-checker.md`（用語集をプロンプトに埋め込み）を検討
- 技術文書で正確性が重要 → `fact-checker.md`（ソース検証に特化）を検討

#### データ分析PJ
- 特定のチャートスタイル・ブランドガイドラインがある → `viz-generator.md` を検討
- 統計的な妥当性検証が必要 → `stats-validator.md` を検討

#### 戦略・企画PJ
- 特定の業界・ドメイン知識が必要 → `domain-expert.md`（業界固有の知識・用語をプロンプトに埋め込み）を検討
- 特定のステークホルダーが固定されている → `stakeholder-simulator.md` のPJ固有版（具体的な人物像・関心事を埋め込み）を検討

### PJ固有エージェントのテンプレート

```markdown
---
name: {name}
description: {このPJで何をするか。自動委譲の判断基準になる}
tools: {必要最小限のツール}
model: {sonnet|opus|haiku}
memory: project
---

{PJ固有のコンテキストとプロンプト}
```

重要:
- `memory: project` を設定し、PJ固有の学習を蓄積させる
- `tools` は必要最小限に絞る（読み取り専用パターン等）
- `description` に自動委譲のトリガー条件を明記する
- グローバルエージェントと名前が被らないようにする

### PJ固有スキルの設計原則

スキルを作成する場合も同じ構造原則に従う:

```
.claude/skills/{skill-name}/
├── SKILL.md          # オーケストレーション（フロー概要、20-30行）
└── reference/        # 詳細データ（テンプレート、チェックリスト等）
    ├── template.md
    └── checklist.md
```

- **SKILL.md は軽量に保つ** — フロー概要とコマンド定義のみ
- **詳細は reference/ に分離** — SKILL.md が肥大化するとコンテキストを圧迫する
- **reference/ はサブエージェントに読ませる** — メインコンテキストを節約

### 非開発PJ向けエージェントテンプレート（参考例）

以下は非開発PJでの活用例。自動生成はしない。実PJで必要が確認されたらユーザー確認の上で作成する。

#### BizDev/営業資料PJ
- `brand-checker.md`: ブランドガイドライン遵守チェック。ブランドボイス、用語、トーンの一貫性を検証する
- skills/create-proposal/: 提案書作成手順のテンプレート

#### ドキュメントPJ
- `terminology-checker.md`: 用語統一チェック。PJ固有の用語集に基づく表記揺れ検出
- skills/review-docs/: ドキュメントレビュー手順

#### リサーチPJ
- `source-validator.md`: 出典品質チェック。ソースの信頼性、引用形式の検証
- skills/research-brief/: リサーチブリーフ作成手順

これらはあくまで参考例であり、PJの実態に合わせてカスタマイズすること。
